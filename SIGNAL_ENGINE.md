# ArduinoWaveHandler Signal Engine

This document defines the internal signal processing and event propagation system used in ArduinoWaveHandler.

The Signal Engine is responsible for handling, routing, and managing all runtime events generated by devices, monitors, and system components.

---

# Overview

The Signal Engine acts as the central event bus for the system.

It handles:

- Device connection signals
- Device disconnection signals
- Serial data signals
- Error signals
- Runtime system signals

All modules communicate through standardized signal packets.

---

# Signal Flow Architecture

```text id="flow001"
DEVICE / MODULE
      ↓
SIGNAL EMITTER
      ↓
SIGNAL ENGINE
      ↓
EVENT ROUTER
      ↓
LISTENERS (console, monitor, logger, runtime)
````

---

# Signal Types

## Core Signals

| Signal     | Description                 |
| ---------- | --------------------------- |
| CONNECT    | Device connected            |
| DISCONNECT | Device removed              |
| DATA       | Incoming serial/data packet |
| ERROR      | System or device error      |
| SCAN       | Device scan cycle           |
| HEARTBEAT  | Keep-alive signal           |

---

# Signal Packet Format

All signals are structured as dictionaries or JSON-like objects.

## Standard Signal Object

```text id="signal001"
{
  "type": "CONNECT",
  "source": "device_manager",
  "target": "monitor",
  "timestamp": 1700000000,
  "payload": {
    "device": "COM3",
    "driver": "CH340",
    "status": "online"
  }
}
```

---

# Emission Rules

* All signals must include a `type`
* All signals must include a `timestamp`
* `source` must identify emitting module
* `payload` is optional but recommended
* Signals must be immutable once emitted

---

# Routing System

Signals are routed based on type:

| Signal     | Route                    |
| ---------- | ------------------------ |
| CONNECT    | monitor, logger, console |
| DISCONNECT | monitor, logger, console |
| DATA       | parser, runtime, logger  |
| ERROR      | logger, console          |
| SCAN       | monitor                  |
| HEARTBEAT  | runtime                  |

---

# Listener Model

Listeners subscribe to signal types:

```text id="listener001"
console → CONNECT, DISCONNECT, ERROR
monitor → CONNECT, DISCONNECT, DATA
logger → ALL
runtime → DATA, ERROR, HEARTBEAT
```

---

# Processing Pipeline

1. Signal emitted by module
2. Engine validates structure
3. Signal queued in event buffer
4. Router distributes signal
5. Listeners process asynchronously
6. Logger records event

---

# Event Buffer

* Stores recent signals
* Default size: 5000 events
* FIFO (first in, first out)
* Used for replay/debugging

---

# Threading Model

* Signals are processed in background threads
* Non-blocking event dispatch
* Safe concurrent queue handling

---

# Error Handling

Signal engine handles:

| Error | Meaning                |
| ----- | ---------------------- |
| SE01  | Invalid signal format  |
| SE02  | Missing required field |
| SE03  | Routing failure        |
| SE04  | Listener crash         |

Errors are logged but do not stop system execution.

---

# Console Integration

Signals directly update live console:

* Device connection updates UI instantly
* Disconnect triggers removal in UI
* Data signals feed parser display
* Errors show alert lines

---

# Monitor Integration

Monitor module uses signals to:

* Track device state changes
* Detect connect/disconnect events
* Maintain snapshot state
* Generate event logs

---

# Logger Integration

Logger receives ALL signals:

* Writes to runtime log file
* Stores structured event history
* Maintains debugging trace
